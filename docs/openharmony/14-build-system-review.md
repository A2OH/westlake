## Executive Summary

The OpenHarmony 4.1 build system is a GN/Ninja-based infrastructure derived from Chromium's build system, significantly customized for multi-architecture, multi-system-level (mini/small/standard) embedded OS development. The system supports arm, arm64, x86_64, riscv64, and mipsel targets using Clang/LLVM toolchains. While the architecture is generally sound and the security hardening posture is above average for embedded systems, this review identifies several critical and high-severity issues related to disabled security features, SSL verification bypass in the supply chain, hardcoded credentials, and build script robustness.

## Build System Architecture

### Overview

The build system has three layers:

1. **Shell/Python entry points**: `build.sh` -> `build.py` -> `hb/main.py` (the "hb" build tool)
2. **GN configuration layer**: `BUILDCONFIG.gn`, `ohos.gni`, `ohos_var.gni`, product/device configs
3. **Ninja execution layer**: Generated by GN, drives actual compilation

**Key files reviewed:**
- `/home/dspfac/openharmony/build/build_scripts/build.sh`
- `/home/dspfac/openharmony/build/build_scripts/build.py`
- `/home/dspfac/openharmony/build/config/BUILDCONFIG.gn`
- `/home/dspfac/openharmony/build/ohos.gni`
- `/home/dspfac/openharmony/build/ohos_var.gni`
- `/home/dspfac/openharmony/build/hb/main.py`

### Findings

#### BS-01: `set -e` immediately negated by `set +e` in build.sh

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, lines 15-16

```bash
set -e
set +e
```

The `set -e` (exit on error) flag is set and immediately disabled on the next line. This means build errors from sub-commands (other than the final exit code checks) will be silently swallowed. The build script does have explicit `$?` checks in some places, but not all command invocations are protected. This can lead to silent build corruption.

#### BS-02: Dual build entry point confusion

- **Severity: LOW**
- **Location**: `build.sh` lines 198-212, `build.py`

The build system supports two code paths: the legacy `entry.py` path (via `using_hb_new=false`) and the new `hb/main.py` path (default). Both are maintained simultaneously, increasing maintenance burden and potential for divergent behavior.

#### BS-03: Excessive use of exec_script in BUILDCONFIG.gn

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/config/BUILDCONFIG.gn`, lines 52, 56, 128, 677

Multiple `exec_script` calls run during GN configuration phase, including `check_mac_system_and_cpu.py`, `check_file_exist.py`, and `run_shell_cmd.py`. Each invocation spawns a Python process during GN evaluation. While documented as a known anti-pattern in the file's own comments (line 282: "Don't call exec_script inside declare_args"), the codebase uses it liberally in non-declare_args contexts. This slows down GN evaluation.

## Product Definitions and Target Configurations

### Architecture

Product definitions follow a layered model:
- **Base configs**: `/home/dspfac/openharmony/productdefine/common/base/` (mini, small, standard system)
- **Inheritance configs**: `/home/dspfac/openharmony/productdefine/common/inherit/` (rich, tablet, 2in1, headless, etc.)
- **Product configs**: `/home/dspfac/openharmony/productdefine/common/products/` (system_arm64_default, ohos-sdk, etc.)

Products inherit from base system definitions and compose subsystems/components declaratively in JSON.

### Findings

#### PD-01: Duplicate component entries in rich.json

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/productdefine/common/inherit/rich.json`

The `security_component_manager` component appears twice (lines 163-166 and 173-176) within the security subsystem. Similarly, `drivers_interface_drm` appears twice (lines 979-982 and 987-990) in the hdf subsystem. The `applications` subsystem also has two separate blocks (lines 736-746 and 757-765), which is semantically invalid JSON for most parsers that treat duplicate keys as override rather than merge.

#### PD-02: Inconsistent JSON field naming

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/productdefine/common/inherit/rich.json`, line 1033

Several entries use `"feature":[]` (singular) instead of the standard `"features":[]` (plural), e.g., `drivers_interface_huks` at line 1033 and `drivers_interface_intelligent_voice` at line 1095. This likely means these entries are silently ignored by the build system.

#### PD-03: Inconsistent JSON formatting

- **Severity: INFORMATIONAL**
- **Location**: `/home/dspfac/openharmony/productdefine/common/inherit/rich.json`

Mixed indentation (tabs vs. spaces) throughout the file, e.g., lines 489, 620, 887, 967, 979. While functionally harmless, it indicates lack of automated formatting enforcement.

## Toolchain Setup and Cross-Compilation

### Architecture

- Primary compiler: Clang/LLVM (prebuilt in `prebuilts/`)
- C library: musl libc (custom sysroot)
- Linker: LLD (LLVM linker) by default
- Supported targets: arm, arm64, x86_64, riscv64, mipsel
- Defined in `/home/dspfac/openharmony/build/toolchain/ohos/BUILD.gn` and `ohos_toolchain.gni`

The `ohos_clang_toolchain` template wraps `gcc_toolchain` and configures target-specific sysroot, CRT objects, and Rust ABI targets.

### Findings

#### TC-01: Hardcoded Node.js version with no update mechanism

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, line 95

```bash
EXPECTED_NODE_VERSION="14.21.1"
```

Node.js 14.x reached End-of-Life on April 30, 2023. This version no longer receives security updates. The build system enforces this exact version with a strict equality check (lines 101-104), preventing developers from using supported Node.js releases.

#### TC-02: Duplicated Rust toolchain configuration

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/toolchain/ohos/ohos_toolchain.gni`, lines 94-104

The `cc_command_args` for Rust compilation is nearly identical across armv7, aarch64, and x86_64 targets. Only minor differences (target triple) exist, but the full string is duplicated 5 times. This is error-prone and harder to maintain.

#### TC-03: LitOS M toolchain lacks security configs

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/config/BUILDCONFIG.gn`, lines 930-934

For `liteos_m` targets, only `stack_protector` is applied. No PIE, no `-Wl,-z,relro`, no `-Wl,-z,now`, and no FORTIFY_SOURCE. While some of these may not be applicable to bare-metal targets, the complete absence of linker hardening flags warrants verification that this is intentional for each flag.

## Component/Subsystem Build Configuration

### Architecture

Components are defined via `bundle.json` files and use templates from `/home/dspfac/openharmony/build/templates/cxx/cxx.gni`:
- `ohos_executable` - executables
- `ohos_shared_library` - shared libraries
- `ohos_static_library` - static libraries
- `ohos_source_set` - source sets

The subsystem registry lives at `/home/dspfac/openharmony/build/subsystem_config.json`, mapping subsystem names to source paths.

### Findings

#### CS-01: Component sanitizer enforcement is vendor-configurable

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/config/sanitizers/sanitizers.gni`, lines 351-405

The `ohos_sanitizer_check` template enforces CFI and integer sanitizer usage for specified parts. However, the enforcement list comes from `ext_sanitizer_check_list_path`, which is vendor-configurable. A vendor can set an empty list, effectively disabling all sanitizer enforcement for the entire system. There is no mechanism to enforce a minimum set of sanitized components.

#### CS-02: Per-component auto-var-init opt-out is unrestricted

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/config/security/security_config.gni`, lines 52-78

Any component can set `auto_var_init = "uninit"` to disable automatic variable initialization, bypassing the security default. There is no allowlist or audit mechanism for opt-outs.

## Security Build Flags

### Hardening Posture Summary

| Feature | Standard System | Lite System | Status |
|---------|----------------|-------------|--------|
| Stack Protector | `-fstack-protector-strong` | `-fstack-protector-all` | ENABLED |
| Stack Protector Return (arm64) | `-fstack-protector-ret-strong` | N/A | ENABLED (if compiler supports) |
| PAC/BTI (arm64) | PAC-RET | N/A | ENABLED |
| PIC | `-fPIC` | `-fPIC` | ENABLED |
| PIE | `-fPIE` / `-pie` | `-fPIE` / `-pie` | ENABLED |
| RELRO | `-Wl,-z,relro` | `-Wl,-z,relro` | ENABLED |
| Bind Now | `-Wl,-z,now` | `-Wl,-z,now` | ENABLED |
| NX Stack | `-Wl,-z,noexecstack` | `-Wl,-z,noexecstack` | ENABLED |
| FORTIFY_SOURCE | Commented out | Commented out | **DISABLED** |
| Auto Var Init | Zero (default off) | N/A | OFF by default |
| CFI | Available, per-component | N/A | Opt-in |
| ThinLTO | Enabled for OHOS | N/A | ENABLED |
| ASLR | PIE enabled (userspace) | PIE enabled | Userspace ENABLED |

### Findings

#### SEC-01: FORTIFY_SOURCE globally disabled

- **Severity: HIGH**
- **Location**: `/home/dspfac/openharmony/build/config/compiler/BUILD.gn`, lines 1006-1016; `/home/dspfac/openharmony/build/lite/config/BUILD.gn`, line 82

```gn
# Need to support fortify ability first in musl libc, so disable the option temporarily
# defines += [ "_FORTIFY_SOURCE=2" ]
```

`_FORTIFY_SOURCE=2` is commented out in **both** the standard system and lite system build configs. The comment says "temporarily," but this has persisted since at least OpenHarmony 3.x. FORTIFY_SOURCE provides runtime buffer overflow detection for common C library functions (memcpy, sprintf, etc.) and is a critical security hardening feature enabled by default in Android, Debian, Fedora, and virtually all modern Linux distributions.

The stated reason is lack of musl libc support. However, musl has partial FORTIFY_SOURCE support, and the lite system configs at `/home/dspfac/openharmony/build/config/compiler/lite/clang/BUILD.gn:49` and `/home/dspfac/openharmony/build/config/compiler/lite/gcc/BUILD.gn:49` **do** define `_FORTIFY_SOURCE=2`, creating an inconsistency.

#### SEC-02: Stack protector disabled for x86 OHOS

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/config/security/BUILD.gn`, lines 66-67

```gn
} else if (is_ohos && current_cpu == "x86") {
    cflags += [ "-fno-stack-protector" ]
```

Stack protection is explicitly disabled for x86 OHOS builds. The emulator platform (`x86_64`) is a primary developer testing environment; disabling stack protectors there means buffer overflow bugs may not be detected during development testing.

#### SEC-03: Auto variable initialization disabled by default

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/config/security/security_config.gni`, line 17

```gn
enable_auto_var_init = false
```

The automatic variable zero-initialization feature (`-ftrivial-auto-var-init=zero`) is available but disabled by default. When enabled, it prevents a class of uninitialized memory disclosure vulnerabilities. Android has enabled this globally since Android 13. The build system has the infrastructure but does not enable it.

#### SEC-04: Hardcoded signing credentials in build configuration

- **Severity: HIGH**
- **Location**: `/home/dspfac/openharmony/build/ohos_var.gni`, lines 303-308

```gn
default_key_alias = "OpenHarmony Application Release"
default_signature_algorithm = "SHA256withECDSA"
default_hap_private_key_path = "123456"
default_keystore_password = "123456"
default_keystore_path = "//developtools/hapsigner/dist/OpenHarmony.p12"
```

Default signing passwords ("123456") and key paths are hardcoded directly in the GN build configuration. While these are clearly development/debug defaults, they are embedded in production build infrastructure without any warnings or compile-time checks to ensure they are overridden for release builds. A production build using these defaults would have trivially compromised application signing.

#### SEC-05: Check list allows disabling PIC/PIE

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/hb/util/post_gn/check_list.json`, lines 36-47

The post-GN checker flags `-fno-PIC`, `-fno-pic`, `-fno-PIE`, `-fno-pie` as disallowed in ldflags and cflags check lists. This is good practice. However, the mechanism is an advisory check rather than a hard enforcement, and the file itself can be modified by vendors.

## Reproducibility and Deterministic Builds

### Findings

#### RP-01: Timestamp macros stripped for non-official builds only

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/config/compiler/BUILD.gn`, lines 257-275

```gn
if (!is_official_build) {
    cflags += [
        "-D__DATE__=",
        "-D__TIME__=",
        "-D__TIMESTAMP__=",
    ]
}
```

The `__DATE__`, `__TIME__`, and `__TIMESTAMP__` macros are blanked out for non-official builds to improve reproducibility. Paradoxically, official builds (where reproducibility matters most for verification) do NOT strip these macros. This is inherited from Chromium's logic where official builds need these for versioning, but for OpenHarmony, official release builds should also be reproducible.

#### RP-02: Build ID uses MD5

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/config/compiler/BUILD.gn`, line 298

```gn
ldflags += [ "-Wl,--build-id=md5" ]
```

The build ID is generated using MD5, which is cryptographically broken. While build IDs are not used for security purposes, using SHA-256 (`--build-id=sha1` at minimum) would be more consistent with security best practices and costs nothing in terms of build performance.

#### RP-03: build.sh prints timestamp into build output

- **Severity: INFORMATIONAL**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, lines 42, 220

The build script prints the current date/time at start and end. This is fine for logging but could affect reproducibility of build logs if those are used for verification.

## Dependency Management and Third-Party Integration

### Findings

#### DM-01: SSL verification globally disabled for npm and ohpm

- **Severity: CRITICAL**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, lines 108, 125

```bash
npm config set strict-ssl false
ohpm config set strict_ssl false
```

SSL certificate verification is unconditionally disabled for both npm and ohpm package managers. This exposes the build to man-in-the-middle attacks where a network attacker could inject malicious packages during the build. This is a supply chain security vulnerability that could compromise any OpenHarmony build performed over an untrusted network.

The npm lockfile is also explicitly disabled (line 109: `npm config set lockfile false`), further undermining supply chain integrity by allowing dependency versions to float between builds.

#### DM-02: npm lockfile explicitly disabled

- **Severity: HIGH**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, lines 109-110

```bash
npm config set lockfile false
cat $HOME/.npmrc | grep 'lockfile=false' > /dev/null || echo 'lockfile=false' >> $HOME/.npmrc > /dev/null
```

The npm lockfile (`package-lock.json`) is explicitly disabled globally and even persisted to the user's `.npmrc`. Lockfiles are a fundamental supply chain security mechanism that ensures reproducible, verified dependency resolution. Without lockfiles, every build can potentially pull different (and possibly compromised) package versions.

#### DM-03: Hardcoded download URL for oh-command-line-tools

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, line 117

```bash
wget https://contentcenter-vali-drcn.dbankcdn.cn/pvt_2/DeveloperAlliance_package_901_9/...
```

The oh-command-line-tools are downloaded via an unversioned, hardcoded URL using `wget` without checksum verification. If this URL is compromised or returns different content, the build tooling could be silently replaced. The download should verify a SHA-256 checksum of the downloaded archive.

#### DM-04: Global npm/ohpm registry configuration modification

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, lines 106-107

```bash
npm config set registry https://repo.huaweicloud.com/repository/npm/
npm config set @ohos:registry https://repo.harmonyos.com/npm/
```

The build script modifies the **global** npm configuration, redirecting all npm operations (not just OpenHarmony builds) to Huawei's mirror. This is a side effect that persists after the build completes, affecting other projects on the same machine. Configuration should be scoped to the build rather than modified globally.

## Build Scripts Quality and Maintainability

### Findings

#### BQ-01: Inconsistent error handling in build.sh

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`

Error handling is inconsistent throughout the script:
- `set -e` is immediately negated by `set +e` (lines 15-16)
- Some commands check `$?` explicitly (lines 148, 164, 214)
- Others do not check return codes at all (e.g., `npm config set` commands, lines 106-110)
- The `wget` command at line 117 has no error checking
- The `init_ohpm` function uses `pushd`/`popd` which can fail silently with `> /dev/null`

#### BQ-02: build.py has a typo in error message

- **Severity: INFORMATIONAL**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.py`, line 82

```python
print("-python-dir parmeter missing value.")
```

"parmeter" should be "parameter".

#### BQ-03: Redundant OS boolean assignments in BUILDCONFIG.gn

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/config/BUILDCONFIG.gn`, lines 531-615

The OS flag assignments (`is_ohos`, `is_linux`, `is_mac`, etc.) are fully duplicated across 7 OS blocks with only one variable differing per block. This could be consolidated using a default-false-then-set-true pattern, reducing over 80 lines to about 20.

#### BQ-04: Redundant condition check

- **Severity: INFORMATIONAL**
- **Location**: `/home/dspfac/openharmony/build/config/compiler/BUILD.gn`, line 301

```gn
if (!is_ohos && !is_ohos) {
```

The condition `!is_ohos` is checked twice (likely a copy-paste error; one should probably be `!is_android`).

#### BQ-05: Source filter logic bug for Mac/iOS

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/config/BUILDCONFIG.gn`, line 644

```gn
if (!is_mac || !is_ios) {
```

This condition uses `||` (OR), meaning the Mac source filter will be applied whenever either `is_mac` is false OR `is_ios` is false. Since a build can only be one OS at a time, this filter is always applied -- even on Mac builds (because `is_ios` would be false). The condition should likely be `&&` (AND): `if (!is_mac && !is_ios)`.

## Developer Experience

### Findings

#### DX-01: SDK auto-build during first compilation

- **Severity: LOW**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, lines 187-194

If the prebuilt SDK is not found, the build script automatically triggers a full SDK build before the main build. This is a significant time cost (potentially hours) that happens silently on first build. While there is a `--no-prebuilt-sdk` flag, the default behavior is surprising and poorly communicated.

#### DX-02: Global system modifications during build

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`

The build script modifies:
- Global npm configuration (registry, SSL, lockfile settings)
- Global ohpm configuration
- `$HOME/.npmrc` (persistent modification)
- `$HOME/.hvigor/` (deletes daemon and wrapper, then recreates)
- `$HOME/.ohpm/` (creates and installs packages)

These modifications persist after the build and affect other projects. Build systems should be self-contained and not modify global user configuration.

#### DX-03: Outdated Node.js version enforcement blocks developers

- **Severity: MEDIUM**
- **Location**: `/home/dspfac/openharmony/build/build_scripts/build.sh`, lines 95-104

The strict Node.js 14.21.1 version check prevents developers with newer (and more secure) Node.js versions from building. Developers must use the exact prebuilt Node.js from the prebuilts directory, which is EOL software.

#### DX-04: Component build template complexity

- **Severity: LOW**

Creating a new component requires understanding the interaction between `bundle.json`, GN templates (`ohos_executable`, `ohos_shared_library`), subsystem config, product config, and the preloader/loader pipeline. The `cxx.gni` template file is nearly 60KB with deeply nested template logic. While powerful, the learning curve is steep.

### Priority Recommendations

1. **Immediate**: Enable SSL verification for npm and ohpm. If corporate proxies require custom CAs, configure the CA bundle rather than disabling verification entirely.

2. **Immediate**: Enable npm lockfiles and commit them. This is a fundamental supply chain security requirement.

3. **Short-term**: Implement FORTIFY_SOURCE support in the musl libc port. This has been "temporary" for multiple releases and represents a significant gap vs. Android/Linux hardening baselines.

4. **Short-term**: Add build-time warnings or assertions when default signing credentials are used in non-debug builds.

5. **Short-term**: Update Node.js to a supported LTS version (20.x or 22.x).

6. **Medium-term**: Enable auto variable initialization (`-ftrivial-auto-var-init=zero`) by default, matching Android 13+ baseline.

7. **Medium-term**: Scope build-time configuration changes (npm registry, ohpm config) to the build directory rather than modifying global user settings.
